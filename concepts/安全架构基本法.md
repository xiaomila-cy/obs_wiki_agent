1 引言

为了提高公司的网络安全防护水平，保障金融信息系统安全稳定运行，积极防范数据安全风险，特制定马上消费金融安全架构基本法（简称“基本法”）。

基本法是马上消费金融安全体系建设的顶层设计蓝图及引领性文件，约定了马上消费金融生产经营活动中必须遵循的信息安全准则。该规范以国家等级保护制度为基础，结合监管标准要求，按照分区分域保护思想，明确约定不同安全域以及区域间的访问控制规则，从宏观层面提出网络安全、主机安全、终端安全、容器云安全、虚拟化安全、应用安全、数据安全、物理环境安全、设备安全、安全运维、安全管理方面的基本安全要求。马上消费金融相关部门需围绕基本法制定具体的安全技术规范、安全技术实施指引以及安全管理实施细则，约定内容不得与基本法中的规定条款相违背。

2 术语解释
- 重要系统：主要指面向客户，涉及账务、交易、风控、运营、催收且时效性要求较高的信息系统。
- IT关键岗位：主要涉及重要系统的运行维护岗，包括基础硬件、系统、网络、数据库、应用维护类岗位。

3 信息安全原则

### 3. 1 “分级保护”原则

应根据信息资产的重要程度及面临的威胁大小等因素决定各类信息的安全保护级别，对不同级别的信息资产实施相适应的保护措施。

### 3. 2 “同步规划、同步建设、同步运行”原则

安全建设应与业务系统同步规划、同步建设、同步运行，避免任何一个环节的疏忽可能给业务系统带来危害。

### 3. 3 “内外并重”原则

安全工作需要做到内外并重，在防范外部威胁的同时，加强规范内外人员行为和审计机制。

### 3. 4 “适度安全”原则

没有绝对的安全，安全和易用性是矛盾的，需要做到适度安全，找到安全性和易用性的平衡点。

### 3. 5 “技术与管理并重”原则

信息安全不是单纯的技术问题，需要在采用安全技术和产品的同时，重视安全管理，不断完善各类安全管理规章制度和操作规范，全局提高安全管理水平。

### 3. 6 “最小授权”原则

公司内受保护的敏感信息只能在一定范围内使用，履行工作职责和职能的内部员工及外部人员在法律和相关安全策略允许的前提下，为满足工作需要，仅被授予其访问信息的最小权限。

4 总体要求

### 4. 1 安全技术架构

### 4. 1.1 网络安全

### 4. 1.1.1 网络安全架构

关于办公域、运维域、业务域、公共域的定义和要求如下：

a. 办公域：主要部署办公性质的信息系统，包括但不限于OA系统、HR系统、采购系统、档案管理系统、财务预算系统、发票管理系统、签章管理系统，此域内的系统停摆或者异常不会影响公司业务域内业务运转。

b. 运维域：主要部署运维监控性质的信息系统，包括但不限于堡垒机、DevOps平台（CICD、ITSM、DBman、CMDB、FASTX）。

c. 业务域：主要部署业务支撑性质且涉及客户或交易的信息系统，包括但不限于账务交易类系统、风控运营类系统、客户营销类系统、客户运营类系统、客户催收类系统。

d. 公共域：主要部署公共支撑组件性质的系统，如SSO统一身份认证平台，此域内系统允许被办公域、运维域、业务域共同调用，但禁止公共域内系统主动访问其他域。办公域的SSO统一身份认证应独立部署。

### 4. 1.1.2 网络访问控制

- 不同逻辑安全区之间应采取必要的隔离措施，安全策略遵循“默认拒绝”、“最小权限”和“授权变更”原则。
- 不同逻辑安全区之间的访问要求精细化至端口级的颗粒度控制。
- 开发测试环境与生产环境必须严格隔离，禁止非授权直接打通开发测试环境与生产环境之间的网络（运维管理、监控数据上报、CI/CD流水线场景例外）。
- 数据区是核心保护安全域，禁止与互联网直连，原则上只接受被动访问，因大数据应用调用配置信息以及KMS加解密的需要，允许数据区主动访问公共组件区。
- 基于公司日常运维监控需要，允许运维区虚拟机管控平台、资源管理平台、监控系统、DevOps平台访问所有的逻辑安全区。
- 办公云是访问业务域的关键通道，必须与互联网完全隔离。如公司风控人员、数据分析人员等核心岗位人员只能通过办公云访问数仓、数据中台、征信、luma、质检等应用系统，运维人员只能通过办公云访问堡垒机进行日常运维操作。
- 公司员工只能通过职场网络和远程VPN接入办公环境，并且只允许在办公环境下访问办公云。
- 公司网络出入向必须实施严格的互联网访问控制策略，其中，生产环境及开发测试环境出向访问默认启用白名单策略，管控颗粒度应精细化至IP级。办公电脑出向网络策略实行分组管理，如开发测试人员共用一套策略，由桌管软件统一实现。
- 默认禁止开发测试环境的互联网访问权限（入向）和访问互联网权限（出向）。如确有联调测试需求，可以通过常规变更流程申请网络出入向访问权限。
- 必须遵循最小化原则对外暴露互联网业务系统的IP和端口。对于供第三方特定机构或用户访问的互联网业务系统，必须对访问源IP地址进行限制，即采取IP白名单策略。
- 金融云的区域划分和区域访问控制遵循上述要求执行。
- 满足上述条款要求的网络策略由公司基础平台部负责，违反上述网络访问规则的网络策略，需经架构、网络、安全、风险条线综合评估，评估通过后可以申请安全例外流程。

### 4. 1.1.3 网络防护体系

- 应在互联网边界部署抗DDoS攻击设备或引入DDoS流量清洗服务，抵御DDoS流量攻击，包括但不限于SYN Flood、UDP Flood、ICMP Flood、HTTP Flood。同时，应在防火墙上设置IP、端口白名单。
- 应在重要网络边界（包括互联网边界、办公至生产边界等）部署Waf等应用安全防护措施，抵御网络扫描、SQL注入、XSS跨站脚本攻击、CSRF跨站请求伪造、命令注入、不安全的反序列化、Struts2漏洞、非法上传下载、恶意爬虫等攻击流量。应用系统发布时，要求接入Waf并启用防护策略，禁止与应用无关的流量通过。同时，应密切跟踪漏洞情报数据，对于新暴露的安全漏洞，应及时更新Waf防护规则。
- 应在网络内部部署可实现入侵检测、网络流量分析、网络流量回溯等功能的安全产品，以识别和监测各类可疑网络攻击行为。
- 应在重要网络边界（包括互联网边界和办公网边界）应用蜜罐与蜜网技术，通过蜜罐识别传统入侵检测手段无法识别的未知攻击。
- 应构建以场景化、实战化、运营化为核心要素的态势感知平台，充分利用大数据、威胁建模、机器学习、威胁情报、安全编排等技术手段进行安全攻击监测预警和安全事件自动化响应。

### 4. 1.1.4 网络安全基线

- 根据网络安全等级保护制度要求和业内最佳实践，规划设计、合理配置网络及安全设备安全基线。基线内容包括但不限于账号管理、口令策略、身份认证管理、账号权限管理、不必要服务的管理、远程访问管理、网络管理、日志管理以及时钟管理等。
- 网络及安全设备基线包括但不限于负载均衡设备、防火墙、路由器、交换机、IDS入侵检测、Web应用防火墙、蜜罐等。

### 4. 1.2 主机安全

### 4. 1.2.1 主机系统安全

- 应基于最小化原则安装主机操作系统。

### 4. 1.2.2 主机镜像安全

- 应弃用不安全的低版本操作系统镜像，同时定期检查操作系统镜像的安全性，持续更新操作系统镜像包。
- 应建立安全可信的内部镜像源/仓库，操作系统、软件安装包等新增镜像必须经过安全检测分析后才能入库。

### 4. 1.2.3 主机入侵防范

- 应建立主机入侵防御体系，对于Web后门、系统后门、暴力破解、异常登录、本地提权、安全漏洞攻击、病毒传播、C&C通信、反弹shell等事件需实时告警并阻断。

### 4. 1.2.4 主机恶意代码防范

- 应实现主机恶意代码查杀能力，确保对主机上的挖矿脚本、勒索病毒、木马后门等恶意程序进行精准识别、告警及自动查杀。

### 4. 1.2.5 主机安全基线

- 根据网络安全等级保护制度要求和业内最佳实践，规划设计、合理配置主机操作系统安全基线，基线内容包括但不限于账号管理、口令策略、身份认证管理、账号权限管理、不必要服务的管理、远程访问管理、网络管理、日志管理以及时钟管理等。
- 主机操作系统基线包括但不限于Linux系统、Windows Server系统。对于AD域控服务器、邮件服务器等安全基线应单独制定。

### 4. 1.3 终端安全

- 应建立完善终端安全防护体系，通过实施组策略、网络准入、桌面管控、防病毒、终端检测与响应、反垃圾邮件等技术措施，结合威胁情报、机器学习、沙箱等手段，多维度感知和管控终端安全风险。
- 应建立终端数据防泄漏体系，落实数据识别、数据标记、数据加密、数字水印、数据访问、数据备份等技术措施，实现数据文件流转全生命周期安全监控。
- 应建立终端上网行为管理体系，通过管控上传下载流量、拦截非法网站、监控数据外发行为、识别员工挖矿行为等措施，规范员工的日常办公行为。
- 应建立终端安全漏洞管理机制，明确漏洞补丁更新方式、频率。
- 应建立终端安全管控措施，对于软件安装、程序运行、即时通讯、网站访问、文件外发、文件打印、USB读写等行为应使用白名单策略。对于非法外联、盗版软件安装、数据窃取等违规行为应借助终端安全防护软件进行监控、阻断和审计。
- 根据网络安全等级保护制度要求和业内最佳实践，规划设计、合理配置终端操作系统安全基线。

### 4. 1.4 容器云安全

### 4. 1.4.1 容器镜像安全

- 应建立内部镜像仓库，并利用镜像签名手段来验证部署到生产环境镜像的完整性。通过与CI/CD流水线工具进行整合，流水线中每个环节完成后需基于镜像hash签名校验，保证镜像的合法性，防止引入非法镜像或被篡改的镜像。
- 应利用镜像扫描工具持续对镜像仓库的镜像和工作节点中运行容器的镜像进行检测扫描，检测镜像是否存在安全漏洞、不安全的配置信息、木马病毒程序、恶意代码以及密钥硬编码情况等。对于不安全的镜像应进行告警或阻断，禁止拉取包含高危漏洞、木马病毒以及敏感信息的镜像进行部署。
- 镜像仓库应实现身份认证和访问权限控制，对于任何镜像的变更或镜像构建操作均需通过认证。
- 应采取加密通信方式（如TLS）访问镜像仓库，防止信息泄露。

### 4. 1.4.2 容器编排及组件安全

- 应基于CNNVD、CVE等漏洞信息库对集群编排工具（如k8s）和容器runtime组件（如docker）进行周期性漏洞扫描，及时发现并修复危险漏洞。
- 应基于合规基线对集群编排工具和容器runtime组件进行周期性扫描，及时发现并整改不符合项。
- 应基于角色权限控制区分平台管理员、集群管理员和租户管理员，并根据业务需求设置合理的权限，禁止未授权访问行为。
- 对于敏感配置信息（如认证密钥，认证用户名、密码）应采用密文方式进行存储，防止未授权访问。

### 4. 1.4.3 容器网络安全

- 应采取IPtables、IPVS、eBPF、Network Policy等方式对集群、节点、容器、命名空间进行有效隔离。
- 应对容器所在的pod网络、Service网络以及节点网络进行安全监控，确保能够及时发现失陷容器，针对暴力破解、反弹shell、Web后门、本地提权、恶意命令执行、容器逃逸等行为，应进行告警。

### 4. 1.4.4 容器安全基线

- 根据网络安全等级保护制度要求和业内最佳实践，规划设计容器运行安全基线。安全基线内容应覆盖主机安全配置、docker 守护进程配置、docker守护程序文件配置、容器镜像和构建文件、容器运行时保护、docker 安全操作、docker集群配置。
- 新创建容器必须通过容器安全合规检查。

### 4. 1.5 虚拟化安全

### 4. 1.5.1 虚拟化网络安全

- 应根据业务需求划分多个vlan或vxlan，并利用虚拟交换机实现不同虚拟机vlan或vxlan间的网络隔离。
- 应引入虚拟化安全产品监控虚拟网络中的数据流，识别和处理来自外部或虚拟机内的异常网络攻击，包括但不限于ARP欺骗攻击、DHCP欺骗攻击、拒绝服务攻击、虚拟机篡改攻击、虚拟机逃逸攻击等。

### 4. 1.5.2 虚拟化主机安全

- 应按照最小必要的原则开放虚拟主机的端口及服务，并基于应用限制进站、出站的网络通信流量，防止恶意代码在网络内部传播和扩散。
- 应对虚拟机镜像进行加密保护，防止非授权访问和恶意篡改。
- 应采取恶意代码检测与防护技术措施，实现对虚拟主机中的后门、病毒、木马、蠕虫、webshell等恶意代码的检测、告警和查杀。
- 应根据公司的漏洞管理要求及时更新虚拟主机漏洞补丁，更新范围包括但不限于esxi主机、vcenter、vsphere client、虚拟机等。
- 应采用多机集群负载技术保障应用系统高可用，规划设计高可用时要避免所有虚拟机处于同一个宿主机上。此外，当宿主机发生意外时，应确保位于宿主机上的虚拟机能够自动迁移至可用宿主机上。
- 应定期创建虚拟机快照和存储快照，以防止数据丢失。

### 4. 1.5.3 虚拟化管理平台安全

- 应仅允许受信的IP地址访问虚拟化管理平台。
- 应对虚拟化管理平台的权限进行精细化划分，避免权限滥用。
- 应对虚拟机的运行日志、访问日志、告警日志以及虚拟化管理平台管理员的操作行为日志进行集中采集管理、审计。

### 4. 1.5.4 虚拟化平台安全基线

- 应根据监管要求以及行业最佳实践制定配套的虚拟化平台（包括虚拟机操作系统、管理平台组件）的安全基线配置规范。
- 应按照安全基线配置规范要求，准确合理地配置安全基线。

### 4. 1.6 应用安全

信息安全控制必须在系统开发周期中适当地构建，包括：

Ø 在需求分析中，确定所需的信息安全措施；

Ø 在应用系统开发、发布、更新过程中，实施安全守则。

### 4. 1.6.1 安全设计

- 采用口令、短信认证码、令牌、证书、生物特征等认证方式对登录用户身份进行鉴别，重要系统[1]登录应采取双因素或多因素方式认证，其中，认证要素不能被轻易劫持、猜解、重置或绕过。
- 按照最小化原则设计用户访问权限，禁止非授权访问。
- 应默认不信任公司内部和外部的任何人和系统，应用需基于身份认证和授权，执行以身份为中心的访问控制和资产保护。
- 对所有输入参数的数据类型、长度、格式、范围和内容进行合法性检查，对所有输出到前端的数据进行安全过滤或者转义编码。
- 采用密码技术手段保证信息传输和存储过程的机密性，其中，互联网业务系统必须采用安全的加密传输通道（如TLS），系统鉴权信息和用户敏感信息必须加密传输、加密存储，如因技术原因或者合作方原因无法统一启用敏感信息加密传输的，必须采用https+ip白名单方式对访问源进行管控。传输关键交易数据应采取数字签名、MAC消息认证码[2]等措施，保证交易不可否认性和完整性。
（6） 对于重要请求及关键交易，系统应具备防重放能力，通过采取请求数量限制、请求频率限制、人机校验等控制措施，防止接口恶意请求。禁止在系统中使用已公开或被已被证实的不安全的加密算法，包括但不限于RC4、DES、3DES、RSA1024、MD4、MD5、SHA1、RIPEMD-160、HMAC-RIPEMD160、PBKDF2-HMAC160，分组密码算法的加密模式不允许使用ECB电子密码本模式。与合作方之间的数据交换与共享禁止使用上述加密算法，在极特殊需求场景下如需使用上述加密算法，必须经过公司风险管理部门的安全评估。
- 禁止系统应用使用SSL的全部版本（SSLv1-SSLv3）以及TLS 1.1及以下版本，推荐使用TLS1.2或以上版本。
- 采用负载均衡、软硬件容错、系统监控等手段保证系统高可用。
- 对系统异常必须进行恰当处理，禁止直接将异常信息反馈给用户，包括但不限于系统详细信息、会话标识、账号信息、调试或堆栈信息、源代码片段等。
- 审计方面，应通过应用层接口将敏感业务的操作日志实时上传，审计日志不应被业务自身删除。
- 禁止将密码、密钥、IP等敏感信息硬编码在系统代码中。
- 禁止在应用日志中打印密码、密钥等敏感信息，日志中出现的姓名、身份证号、手机号、银行卡号等客户敏感信息必须按照公司统一的标准规范进行脱敏或加密。
- 禁止在系统中引入不安全的组件、框架与代码。
- 除上述要求，新开发系统或采购系统还必须满足公司的开发安全规范要求。
- 应用系统开发全生命周期活动过程涉及开发环境（dev）、测试环境（test）、预发布环境（stage）和生产环境（prod）。为了保证开发安全，不同环境需要进行逻辑隔离或物理隔离，严禁在开发环境和测试环境中使用真实（生产）数据。

### 4. 1.6.2 安全评审

- 建立安全评审工作机制，并按照相关应用安全规范开展安全评审工作，安全评审记录应存档。

### 4. 1.6.3 安全编码

- 公司技术部应结合业内最佳实践，建立可落地的通用安全编码规范，包括Web安全编码规范、APP安全编码规范。
- 研发人员应根据Web安全编码规范、APP安全编码规范进行安全编码，可以利用开发辅助插件识别代码中的安全漏洞，避免将漏洞写入系统中。
- 研发人员必须使用安全可信的平台/工具开发应用程序。

### 4. 1.6.4 安全测试

- 在代码提交至代码仓库时，应进行自动化代码安全扫描，对于确实存在的安全缺陷，需提前修复。其中，代码安全扫描规则库必须定期优化和更新，以减少误报和漏报。
- PPR应用版本上线前必须通过安全漏洞扫描和人工渗透测试。
- 对外发布的APP客户端程序、API接口和SDK包，应通过专业的第三方机构或平台进行安全检测评估。

### 4. 1.6.5 安全部署

- 应用上线部署前，应删除相应的测试内容，包括但不限于测试页面、测试用例、测试代码、测试接口等。应用上线部署后，应删除默认部署页面，禁止留存SVN/Git相关文件、备份文件等。
- 对外发布的APP客户端程序和SDK包，应使用代码混淆、安全加固、调试器检测等手段进行安全保护，避免程序被逆向破解和恶意篡改。

### 4. 1.6.6 安全防护

- 对外发布的系统或API接口，必须接入网络安全防护体系中统一保护。
- 根据业务重要性，合理引入RAS动态应用防护技术产品，通过动态封装、动态验证、动态混淆、动态令牌、动态挑战手段防护异常登录、身份仿冒、撞库、bot爬虫、刷单等恶意行为。

### 4. 1.7 数据安全

### 4. 1.7.1 数据采集安全

- 涉及客户个人信息的，应按照明示同意、最小必要等原则，通过隐私政策及用户协议对用户进行明确告知，并通过信息加密、展示屏蔽等技术手段，保障收集过程中的安全性。
- 数据采集后，应按照统一的规范进行数据标准化处理，统一格式后再入库，并要求标注数据来源，以供未来数据回溯分析。

### 4. 1.7.2 数据传输安全

- 在互联网传输数据时，应采用通讯加密技术对数据通道进行保护（如专线、vpn或https），禁止使用未加密的通讯技术传输数据。
- 与第三方之间的数据传输应通过双方约定的系统接口或受控的数据集中传输平台进行，防止被非授权第三方截取、篡改。暴露接口应设置网络访问IP白名单策略。同时，接口访问必须具有鉴权机制和恶意调用监测防范机制，包括但不限于限制接口请求频率、限制单次/单日接口请求量级，以规避来自合作方的批量撞库和爬虫风险。
- 禁止数据跨境传输。

### 4. 1.7.3 数据存储安全

- 姓名、身份证号、手机号、银行卡号个人信息四要素、密码应进行加密存储，采用的加密算法、密钥长度及密钥管理方式应符合国家密码管理部门与行业主管部门要求。
- 个人信息保存期限应遵循国家个人信息保护法律法规要求和行业主管部门有关规定要求，同时应符合客户授权使用的目的所必需的最短时间要求。
- 公司技术部应制定数据备份策略，明确备份内容、备份方式、频率以及保留周期等内容，并定期对数据进行备份和恢复性测试。
- 公司数据只允许存储在中国境内，并存储在公司自有设备中，禁止存放在公有云环境。

### 4. 1.7.4 数据处理安全

- 公司通过办公云对数据清洗转换等加工操作进行数据隔离管理，员工只允许通过办公云访问敏感应用系统，只允许通过办公云下载ftp/ftps/sftp的数据文件，并在办公云系统完成日常数据分析工作，经过审批授权，可以借助数据交换平台在办公云、生产系统、办公终端之间安全摆渡数据。
- 公司应按“最小必要”原则赋予数据使用者完成工作所需的最小访问权限。
- 公司内部业务数据库以及大数据平台的敏感数据必须进行加密存储，未经审批授权，禁止任何人提取相关数据。
- 公司应建立数据脱敏标准和敏感数据提取流程，以规范内部数据使用。通过应用系统查询和导出敏感数据需基于角色进行脱敏，因业务办理场景需要（如客诉处理、客户关怀）需查看敏感信息时，可采取点击查看或授权查看方式进行处理，但必须做好操作痕迹记录。用于开发测试的数据必须进行脱敏处理后才能使用。
- 公司应建立数据安全访问控制措施，对于违规查询、非法下载数据的行为应进行有效审计和预警。

### 4. 1.7.5 数据交换安全

- 同合作方进行数据交互时，必须考虑数据合规风险，数据交互方案需提交合规、法务及信息科技风险部门审核。
- 为了保障数据安全，原则上应对姓名、手机号、银行卡号、身份证号、地址等交互敏感数据进行加密或哈希处理，对于公司内部系统之间的文件交换场景，应实现文件内容加密。
- 与第三方机构存在撞库或联合建模需求的场景，宜优先使用隐私计算技术保障用户的数据安全，因业务发展需要与外部第三方共享敏感数据时，应与第三方签订数据安全保密协议。

### 4. 1.7.6 数据销毁安全

- 应建立数据销毁策略和制度，明确销毁对象、流程、方式和要求。
- 数据销毁流程应经过登记和审批，数据销毁过程在监督下执行，采用消磁、粉碎等方式，应确保介质中的敏感数据不可再被恢复利用。如无条件做到消磁或粉碎介质，应至少采用具有擦除文件功能的系统命令或工具对相应介质进行多次擦除。

### 4. 1.7.7 数据管理安全

- 应根据数据的重要程度，对数据进行分类分级，明确敏感数据定义，建立数据安全管理全生命周期管理技术措施。
- 敏感业务部门宜部署DLP数据防泄漏产品和DRM数据版权管理产品，防止公司关键资产（如数据、代码、文档）泄漏。

### 4. 1.8 物理环境及设备安全

### 4. 1.8.1 物理环境安全

- 应采取相应的物理安全防范（如监控系统、门禁系统、防盗报警系统），以防止公司信息资产及信息系统在未经授权下受到物理访问、破坏或干扰。
- 应设计和实施适当的物理环境保护措施，防范或减低火灾、水灾或人为灾难对信息设备的影响。

### 4. 1.8.2 设备安全

- 关键网络设备、服务器、存储设备、传输链路等应采用双链路/多链路、双机备份、集群、Raid等方式规避单点故障，保证业务连续性。
- 核心交换机、路由器、防火墙、服务器等关键设备的配置文件应定期备份，并要求在多数据中心同步备份存储。
- 核心交换机、路由器、防火墙、服务器等关键设备应配备备品备件，以提前预防随时可能发生的生产事故。在无备件的情况，必须与合作方明确具体的应急响应服务时间，以保障公司业务连续性。

### 4. 2 安全运维架构

- 公司安全运维体系建设必须遵循以下原则：

l 认证和授权原则：在访问公司的信息和信息系统前，必须对其身份进行认证，访问控制权限必须遵循“权限最小化原则”，只开通必要的权限。并按照“除非明确允许，否则一律禁止”的原则来设置访问控制规则。

l 职责分离原则：不容许单独1人进行高风险操作，相关操作流程必须采取有效的监控措施，如分拆操作流程并由两个或以上人员分别执行；轮流由不同人员操作；强制执行安全审查及审批程序。
- 公司技术部应建立安全运营常态化工作机制。指定专人对安全产品进行运维，保证产品补丁库和规则库保持最新，同时对漏洞攻击、钓鱼攻击、APP仿冒、内容安全、数据泄露、恶意程序感染等事件进行持续监测和报告。此外，禁止为个人域账号与相关系统的管理员账号（如AD域、邮件系统、补丁服务器）设置相同的账号及密码。
- 公司技术部应建立安全漏洞管理规范，明确漏洞信息获取、影响评估、分类分级、闭环处置等环节的安全管理要求。
- 公司技术部应建立安全监控体系，通过建设统一的监控平台，合理设置监控预警参数，实现安全事件及时预警。
- 公司技术部应建立健全应急灾备体系，通过打造多活数据中心，保障业务连续性。同时，应按照公司日志管理规定留存相关日志（包括网络设备、安全设备、操作系统、数据库、中间件、应用系统等），保存时间应不少于6个月。

### 4. 3 安全管理架构

- 人员安全：所有工作岗位必须具有安全职责描述，并明确说明岗位的敏感性。员工在入职前必须通过审查，并签订安全保密协议。人员岗位发生变化或离职时，必须执行系统账号权限回收、变更或注销程序，确保信息资产保护不受影响。IT关键岗位[2]人员必须设置AB角。违反公司信息安全规范的人员将按处罚规定予以处理。
- 安全意识：所有员工必须通过接受信息安全教育、培训来提高信息安全意识，确保所有员工认识到信息安全的重要性，明白各信息安全标准的必要性及了解自己所需承担的信息安全责任，并遵循公司信息安全相关规定。
- 角色及责任：各部门所担当的信息安全保护角色及其所需承担的责任必须明确定义，确保各部门人员知晓其职责并落实信息安全工作。
- 合规安全：应严格遵循国家法律、监管法规、行业规范守则的信息安全要求，并以最高规范为实施原则。
- 隐私安全：应根据法律、法规、合同要求，保护客户信息及隐私。
- 第三方服务管理：外包服务并不能免除公司的信息安全保护责任。在选择、评审外包服务供应商及商议服务合同时，应充分考虑信息安全标准。在外包服务实施后，应定期审查外包商的业绩标准。
- 采购管理：应确保公司信息安全产品的采购和使用符合国家相关管理部门的规定，应确保密码产品与服务的采购和使用符合国家密码管理主管部门的要求。
